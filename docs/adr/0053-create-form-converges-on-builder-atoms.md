# New Booking form converges on the Booking Builder's section atoms

**Status:** accepted — extends the [[Booking Builder]] (PRD #511, Module B/E) by reviving create-side convergence that was deliberately descoped during the original build. Supersedes #537 (its People/Venue scope becomes the first slice here).

The [[Booking Builder]] shipped one self-contained editor *atom* per concern — Overview, People, Venue, Details, Itinerary, Music — composed by the Builder one-pager and the per-card quick-tweak sheets in a **self-saving** regime (the atom owns its `PATCH`). PRD #511 also promised each atom a **controlled** regime (create mode — value bubbles to the atomic `POST`, the atom does not save) and that the New Booking form would be built from the same atoms.

That create-side half was **deliberately descoped** during implementation. #525's acceptance criteria froze it explicitly — *"creation flow unchanged… only People consolidates to one block"* — to protect Story 39 (the lean, fast enquiry-capture flow) and keep #525 focused on the commit checkpoint + drawer retirement. Even the People-consolidation was then split out into #537. The result on `main`: the New Booking form still renders three bespoke legacy inline blocks (`InlineContactBlock` / `InlineVenueBlock` / `InlineAgentBlock`) plus raw basics fields — a **parallel implementation** of "set up a booking" that visibly diverges from the Builder's first part. Fixing only People/Venue (#537) would leave a half-atom / half-legacy form: partial convergence does not remove the divergence the user actually feels.

**Decision:** converge the New Booking form onto the Builder's section atoms **comprehensively** — every create-form concern renders the *same atom* as its Builder counterpart, so the two surfaces are identical section-by-section. Specifically:

1. **Atom-level, not shell-level.** Create keeps its own **lean, single-submit, required-gated form-shell** ending at the atomic `POST` + commit checkpoint (ADR-0047); the Builder keeps its sectioned rail/stepper workspace. The duplication dies at the *atom* layer (one implementation of "edit people", "edit venue", etc.), while the two shells stay deliberately distinct. **Shell-chrome convergence** — running the create flow *on* the Builder shell so there is literally one continuous surface — is **explicitly deferred** as an independent future decision; it does not block this work, and the cores extracted here are exactly the groundwork it would need.

2. **Mechanism: extract a controlled presentational core per atom.** Following the established `DetailsFields` / `MusicFields` / `ItineraryFields` precedent, each atom's pick/edit core is extracted as a presentational component that bubbles its selection via `onChange` (no booking `id`, no `PATCH`, no save row). `PeopleAtom`'s inner `RoleField` already is such a core; `VenueFields` and an Overview core are to be extracted. The self-saving atom and the create form become two thin compositions of the same core. This is **not** a `mode` flag bolted onto the atom.

3. **Story 39 preserved by the lean-variant rule.** Create-mode variants render the **same field set as today** — no heavier. Packages is **template-multiselect only**; Music is **on/off**; the full set-building / logistics editors stay **post-commit**. Convergence is de-duplication + visual consistency, never dragging a post-commit editor into create.

4. **No backend change.** New contacts/venues continue to **eager-create at submit** — the atom bubbles an existing-or-new selection, and the create shell resolves `new → id` via `POST /contacts` before the booking `POST`, exactly as the inline blocks do today. The atomic `POST /bookings` stays FK-only (ADR-0047 intact); nested new-contact creation is **not** introduced.

5. **Status stays create-shell-owned and becomes a coaching control.** Status is **not** an atom field — preserving Overview's deliberate invariant that the status transition is a standalone confirmed action, never a field. In create, the shell renders status as **radios over the forward statuses** (`Enquiry`, `Provisional`, `Confirmed`, `Ready`, `Complete`; `Cancelled` excluded), each surfacing its CONTEXT-canonical meaning (in-context teaching, per the *discoverability is in-context* principle) from a shared `STATUS_DESCRIPTIONS` constant. Default pre-selection remains `[[UserProfile]].defaultBookingStatus`. (`Ready` is a legitimate creation status — e.g. a series gig needing no pre-ready prep; `Complete` supports backfilling an already-played gig.)

6. **Retire the legacy inline blocks** (`InlineContactBlock` / `InlineVenueBlock` / `InlineAgentBlock`) once nothing else consumes them; `VenueAtom` already meets or exceeds their field capture (Google Places + parking/access/equipment).

**Considered and rejected:** *Partial convergence (People/Venue only, #537 as written)* — leaves a half-atom form, i.e. the very divergence the work targets. *Shell-level convergence (create runs on the Builder shell)* — deferred, not rejected: a larger refactor that fights the lean-form intent; revisit if section-level sameness turns out not to satisfy. *Nested new-contact creation in the atomic `POST`* — contradicts ADR-0047's FK-only design, and the orphan-contact-on-abandon it would prevent is not a felt pain today. *Folding status into the Overview atom* — breaks the "status is never a field" invariant and the standalone-confirmation transition.

**Consequences:** the New Booking form and the Builder stop diverging — one set of section editors, composed by two shells. This **reverses the #525 "creation unchanged" descope on purpose**, with Story 39 protected by the lean-variant rule (item 3) rather than by leaving the form bespoke — a future reader comparing the rebuilt create form against #525's "creation unchanged" AC has the why here. The status control gains a small, intentional enrichment (coaching radios) over today's plain dropdown — a justified deviation from "identical fields" in service of discoverability. The work is a tracking-issue-sized effort sliced per concern (People+Venue → Overview/basics+series → Status coaching → Packages → Music → Notes), with #537 absorbed as the first slice. CONTEXT needs no change: the status canon (lifecycle definitions) already lives there.
