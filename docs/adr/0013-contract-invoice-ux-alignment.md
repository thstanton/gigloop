# ADR-0013 — Contract and Invoice UX Alignment

## Status
Accepted

## Context

The contract and invoice workflows had diverged in the admin UI:

- **Invoices** had a clear lifecycle: create draft → send → (optionally) mark paid. The draft state was editable and visible in the admin before anything was sent to the client.
- **Contracts** had no draft state, no preview, and no dedicated admin section. The musician had no way to view or customise the contract content before sending it. Once sent, the contract was visible to the client via the portal but invisible to the musician until signed. The signed PDF only appeared after the client completed the signing flow.

This asymmetry meant musicians could accidentally send an unchecked contract, had no recovery path if the wrong template was applied, and had no visibility into what was live on the portal.

The additional problems identified:

- The `contractSignedAt` field was the only contract-related state on a Booking — no representation of the "sent but not yet signed" phase.
- The checklist item "Send contract/deposit email" was reachable without any precondition on the contract content.
- Invoices had a visual alignment problem: the "Mark paid" action pushed the status pill off its consistent left-aligned position.
- The "Documents" section title was ambiguous — it grouped invoices, signed contracts, and song lists without a clear organising principle.

## Decision

### Contract lifecycle

Contract state is fully **derived** from existing booking fields — no new `contractStatus` enum. Four states:

| State | Derived from |
|---|---|
| Not started | `contractContent` is null |
| Draft | `contractContent` non-null AND no sent contract email |
| Sent | A `contract_cover` or `contract_and_deposit_cover` `Communication` with `status = SENT` exists |
| Signed | `contractSignedAt` is set |

### `Booking.contractContent`

A new nullable Tiptap JSON field stores the per-booking contract content. It is populated by the "Create contract" action and is the authoritative source for what the client reads on the portal — the contract template is no longer re-rendered at portal time.

**Variable substitution at creation time:** when "Create contract" is triggered, all template variables are substituted as plain text and stored in `contractContent`. No variable chips survive into the booking record. This means the stored content is the exact text that will be shown to the client. The musician can then edit it before sending.

**Template selection:** while only one contract template exists, "Create contract" uses it directly without a picker. A picker will be introduced when multiple contract templates exist.

**After sending:** `contractContent` becomes the contract the client signs. The Tiptap editor transitions to read-only mode.

**After signing:** the signed PDF (generated by the portal signing flow) becomes the source of truth. The Tiptap content is retained but the PDF is the canonical document.

### Contract edit sheet

A sheet component (analogous to the invoice create/edit sheet) hosts a Tiptap editor pre-loaded with `contractContent`.

- **Draft state:** editor is fully editable. Opened by the Edit (pencil) icon on the contract row.
- **Sent/Signed state:** editor is read-only. Opened by the Preview icon on the contract row. This ensures the musician can always view the contract content regardless of state — there is no phase in which the contract is visible to the client but not the musician.

### Right panel section order

Flat card sequence — no grouping header between finance cards:

1. Checklist
2. Contract *(new)*
3. Invoices
4. Documents
5. Communications

### Checklist gate

The "Send contract/deposit email" checklist item is **visible but blocked** until `contractContent` exists on the booking. A hint explains the precondition: "Create a contract first." This preserves the musician's awareness of the step while preventing premature sends.

### Invoice row action alignment

Actions are rendered as icon buttons **before** the status pill. The pill is always the rightmost element. This preserves the left-border visual rhythm established in ADR-0012.

Row layouts by state:

- **Draft:** `[amount] [Send ✉] [Edit ✏] [pill]`
- **Sent (not overdue):** `[amount] [Mark paid ✓] [pill]`
- **Sent (overdue):** `[amount] [Mark paid ✓] [pill (overdue derived state)]`
- **Paid:** `[amount] [pill]`

A download icon is added to Sent and Paid rows to access the invoice PDF from R2.

### Download links in cards

Each card surfaces its own download link rather than routing through a central Documents section:

- **Contract card:** signed contract PDF download (visible once `contractSignedAt` is set)
- **Invoices card:** invoice PDF download on each Sent/Paid row
- **Music Form section:** song list PDF download (visible once `MusicFormResponse` exists)

Documents section remains for the song list and any future document types — it is not removed.

## Alternatives considered

- **`contractStatus` enum stored on Booking:** Rejected — state is fully derivable from existing fields. A stored enum would require keeping it in sync and would be redundant with `contractContent`, sent Communication records, and `contractSignedAt`.
- **Re-render contract template at portal time:** Rejected — the musician's per-booking edits would be lost. Storing `contractContent` at creation time is the only way to honour edits.
- **Preview icon only in Sent state:** Rejected — the musician would have no way to view the contract during the window between creation and sending. The edit sheet serves as preview in Draft state; the read-only preview sheet is shown in Sent and Signed states.
- **"Mark paid" as a right-aligned action:** Rejected — right-aligning actions pushes the status pill out of its consistent vertical alignment. Icon buttons before the pill preserve the ADR-0012 pattern.
- **Remove Documents section:** Rejected — the song list PDF has no natural home in the Music Form section (the card is not finance-adjacent). Documents section stays as a general download hub; individual cards also carry their own download links for convenience.

## Consequences

- `Booking.contractContent` (nullable Tiptap JSON) requires a Prisma schema change and migration.
- The portal's `GET /booking/:token/contract` endpoint must now read from `Booking.contractContent` rather than re-rendering the contract template. If `contractContent` is null, the endpoint returns 404.
- A new `POST /bookings/:id/contract/create` endpoint performs the variable substitution and stores the result in `contractContent`.
- The `BookingChecklist` "Send contract/deposit email" item requires a new precondition check: `contractContent !== null`.
- The `BookingChecklist` entry in CONTEXT.md must be updated to reflect the new "Create contract" item and the gate.
- `ContactEditSheet` pattern (ADR-0012) is the model for the contract edit sheet — controlled open/close state, no URL params.
- Invoice row components require a layout change to move action icons before the pill.
