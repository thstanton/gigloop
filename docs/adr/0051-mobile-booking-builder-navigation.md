# Mobile Booking Builder navigation — ambient-primary, tab-bar takeover

**Status:** accepted — refines the mobile-nav assumption recorded during the Booking Builder design (PRD #511).

The [[Booking Builder]] is a single scrolling one-pager stacking the concern atoms in spine order, with a desktop **completeness rail** alongside (vertical list, per-section status icons, jump-to-section). The original design note assumed that *on mobile the exit-backstop would become the primary safety net* (the rail "can't co-exist with editing" at 375px). When the one-pager shipped (#524), the rail was simply `hidden md:block` on mobile and the exit-backstop fired only from a page-end "Done" button — which the three real mobile exits (the top "Back to booking" link, OS back-swipe, and the persistent AppShell bottom tab bar) all bypass. So the backstop caught almost nobody: the stated principle was false as built.

**Decision:** on mobile, **ambient progress carries the safety net, not the exit-backstop** — inverting the original assumption. Concretely:

1. **Tab-bar takeover.** Inside the Builder, the AppShell bottom tab bar is hidden on mobile. This is a deliberate exception to the otherwise-universal mobile bottom nav: the Builder is a sustained, focused editing task, not a place to tab-hop out of, and hiding the bar removes one un-guardable exit. The top app bar stays.
2. **Ambient-primary.** A progress affordance is visible throughout editing, so leaving a booking incomplete is never a surprise — which fits the Builder's incremental, stage-agnostic ethos (a booking is *legitimately* incomplete most of its life). The exit-backstop is demoted to a gentle, dismissible confirm. The ambient signal is deliberately narrow: only the three concerns with a detectable done-state — **People, Venue, Itinerary** — report completeness (mirroring `buildCompletenessMap`, and the same predicates that drive the structural checklist items). Those are the load-bearing "is this booking set up" signals, so three live signals are enough to earn the takeover; the other five concerns have nothing to report and make no completeness claim.
3. **Top sticky stepper.** The mobile affordance is a horizontal stepper at the top (`md:hidden`; desktop keeps the vertical rail): one node per spine section, all eight navigable. The three status-bearing sections reuse the *same* completeness icons as the desktop rail; the other five render as plain position dots (no icon — nothing to claim). The active node is highlighted and the current section's name shown as text, both synced to scroll position. Tapping a node jumps to that section. The stepper *is* the at-a-glance — there is no expand-to-sheet.
4. **Asymmetric exits.** The page-end **Done** is a completion declaration and fires the demoted backstop only when sections are still empty; the top **Back** link and OS **back-swipe** are always clean exits with no gate (gating leave-for-now would nag the very flow the incremental design protects).

**Considered and rejected:** keeping the Builder in-shell with the tab bar (leaves the un-guardable tab-bar exit and a contested bottom edge); making the exit-backstop reliably fire on every exit (gates a flow designed to be left incomplete, and back-swipe leaks past it regardless); a bottom progress bar (Tim preferred the top, where the stepper can mirror the desktop icons directly).

**Consequences:** AppShell must conditionally drop the bottom tab bar on the Builder route — the first surface to hide it. The orchestration trio (ambient / positional / exit-backstop) maps onto mobile as: ambient = the completeness icons on the three status-bearing sections (People, Venue, Itinerary); positional = the active node + label (carried by all eight); exit-backstop = the Done confirm (which already lists only those empty status-bearing sections). The earlier "exit-backstop = primary mobile safety net" framing is reversed here, so a future reader finding the inverted behaviour in code has the why.
